
# Release 2.1.2 (2026-06-24)

## Bug fixes

- **triage 42 new CVEs blocking the 2.1.1 prod publish (#351)**
* fix(trivy): triage 42 new CVEs blocking the 2.1.1 prod publish

The 2.1.1 release CD failed: all 13 prod image-publish jobs hit the Trivy CRITICAL/HIGH gate on a fresh batch of 42 CVEs not yet in .trivyignore. None is exploitable in the images' actual runtime usage.

Add all 42 under grouped, documented rationale blocks:
- 17 linux-libc-dev kernel-header false positives (incl. the lone CRITICAL, CVE-2026-43185) — containers use the host kernel.
- 19 Go stdlib/module CVEs (x/net, x/crypto, containerd, gh, stdlib) embedded in prebuilt tool binaries — static linters / known-host API clients that do not exercise the affected paths.
- 3 Debian no-fix packages (jq, libsqlite3-0) — not run on untrusted input.
- 3 Debian fix-available packages (libpython3.13, openssl) on the stale-base ruby:3.2 / rust:1.92 / rust:1.93 images — clears on base refresh.


## Features

- **add OpenTofu to the dev-base image (#353)**
Add a pinned OpenTofu 1.12.3 release binary to the dev-base image via a new common/opentofu.dockerfile fragment, mirroring the security-tools tarball-plus-checksum pattern. This enables in-sandbox tofu fmt -check and tofu init -backend=false && tofu validate for OpenTofu modules with no cloud credentials, unblocking vergil-vm#250.

OpenTofu release artifacts name the architecture exactly as TARGETARCH (linux_amd64 / linux_arm64), so no arch translation is needed and both target architectures are covered.
